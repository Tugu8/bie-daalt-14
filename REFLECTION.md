# Эргэцүүлэл — Бие даалт 14

## 1. Аль assertion хамгийн их үнэ цэнэтэй санагдсан вэ? Яагаад?

Chain дээр суурилсан assertion хамгийн их үнэ цэнэтэй санагдсан. `Create Task` request-д `pm.environment.set("taskId", d.id)` ашиглан дараагийн `Update Task` болон `Delete Task` request-д автоматаар taskId дамжуулах нь реал ертөнцийн ашиглалтыг тусгасан. Нэг request-ийн гаралт нөгөөгийнхөд оролт болдог энэ зарчим нь integration testing-ийн гол давуу тал юм. Хэрэв chain байхгүй бол Update, Delete тест бичихэд ID-г гараар тохируулах шаардлагатай болох байсан — энэ нь автоматжуулалтын үндсэн зорилгыг алдагдуулна.

Мөн `Response time < 1s` assertion бодитой үнэ цэнэтэй. Яагаад гэвэл зөвхөн "хариу ирлээ" гэдгийг биш "хариу хурдан ирлээ" гэдгийг шалгадаг. Серверийн гүйцэтгэлийн асуудлыг эрт илрүүлэх боломж олгодог.

## 2. Negative test жишээгээ дэлгэрэнгүй тайлбарла — яг ямар алдааг олох вэ?

`GET /api/tasks/999999` — байхгүй ID-тай task хайх request. Энэ тест хоёр зүйлийг шалгана:

**Нэгдүгээрт:** Сервер `404 Not Found` status буцааж байна уу? Зарим хөгжүүлэгчид байхгүй мэдээлэл хайхад `200 OK` буцааж, хоосон объект өгдөг — энэ нь буруу. HTTP стандартаар 404 буцаах ёстой.

**Хоёрдугаарт:** Хариу дотор `error` field байна уу? Frontend хэрэглэгчид алдааны мессеж харуулахад энэ field шаардлагатай. Хэрэв `error` field байхгүй бол клиент ямар алдаа гарсныг мэдэх боломжгүй болно.

Энэ тест нь "хэвийн ажиллах"-ыг шалгах хангалтгүй — алдааны зам ч бас contract гэдгийг харуулдаг.

## 3. Postman дотор амжилттай ажиллаж байсан тест Newman-д орвол fail болсон уу? Яагаад?

Тийм, нэг удаа тохиолдсон. Collection Runner-д `Empty request URL` алдаа гарсан. Шалтгаан нь request-уудыг Postman-д **хадгалаагүй** байсан — draft байдлаар нээлттэй байсан request-ууд collection-д бичигдээгүй учраас Newman тэдгээрийг олоогүй.

Засал: request бүрийг `Ctrl+S`-ээр хадгалсны дараа дахин ажиллуулахад бүх тест амжилттай болсон. Энэ туршлагаас "хадгал, дараа нь ажиллуул" гэсэн дүрмийг сурсан.

## 4. Token эсвэл secret-ыг хэрхэн зохицуулсан вэ?

Энэ API auth шаардахгүй тул token байгаагүй. Гэсэн ч environment variable-ийн соёлыг бүрэн дагасан:

- `baseUrl`-г `env.dev.json` болон `env.ci.json`-д тусад нь хадгалсан
- Collection дотор URL-г hardcode хийгээгүй — бүгд `{{baseUrl}}` хувьсагч ашигласан
- `env.dev.json`-д бодит утга, `env.ci.json`-д CI-д тохирсон утга тавьсан

Token шаардсан API сонгосон байсан бол `env.dev.json`-д `REPLACE_THIS_TOKEN` placeholder ашиглаж, файлыг `.gitignore`-д оруулах байсан. GitHub Secrets ашиглан CI-д дамжуулах байсан.

## 5. API өөрчлөгдвөл collection-ийн аль хэсэг хамгийн их эвдрэх вэ?

Response schema-г шалгадаг assertion-ууд хамгийн эмзэг байна. Жишээ нь:

- `data.title` → `data.name` болж өөрчлөгдвөл `Update Task`-ийн `"Title updated"` тест fail болно
- `{ data: {...} }` бүтэц → `{ result: {...} }` болвол бараг бүх тест нэгэн зэрэг fail болно
- Status code өөрчлөгдвөл жишээ нь POST 201 → 200 болвол `Create Task` fail болно

**Эмзэг газрыг бууруулах арга:** JSON Schema validation ашиглах — `pm.response.to.have.jsonSchema(schema)` тушаалаар бүтэн схемийг нэг дор шалгавал аль field хаана эвдэрснийг тодорхой харуулдаг. Мөн `baseUrl`-г environment-д хадгалдаг учраас сервер шилжих үед зөвхөн нэг газар засахад хангалттай.
